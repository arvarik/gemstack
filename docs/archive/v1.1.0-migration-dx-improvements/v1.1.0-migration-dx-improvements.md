## Phase A: Core Parser Fixes (single PR)

> These three are tightly coupled and fix the root cause: **the tools generate so much noise that users learn to ignore them.**

### Task A1: Section-Scoped Dependency Extraction

**File:** [`differ.py`](file:///Users/arvind/Documents/github/gemstack/src/gemstack/utils/differ.py)
**Method:** `_extract_documented_deps()` (lines 221–237)

1. Add a `_extract_section()` helper that parses markdown into `{heading: content}` by `##` boundaries.
2. Rewrite `_extract_documented_deps()` to only extract from sections whose headings match: `Tech Stack`, `Dependencies`, `Core Dependencies`, `Core Libraries`.
3. Keep the existing backtick regex within those scoped sections.
4. Filter out candidates that look like: versions (`>=3.10`), SQL types (`BIGINT`), error codes, or JSON tags.

### Task A2: Section-Scoped Env Var Extraction

**File:** [`differ.py`](file:///Users/arvind/Documents/github/gemstack/src/gemstack/utils/differ.py)
**Method:** `_extract_documented_env_vars()` (lines 254–268)

1. Reuse the `_extract_section()` helper from A1.
2. Only extract ALL_CAPS backtick items from sections matching: `Environment`, `Configuration`, `Config`.
3. Existing regex + `re.match(r"^[A-Z][A-Z0-9_]{2,}$")` filter stays, but scoped to the right sections.

### Task A3: Validator Stale Reference Scoping

**File:** [`validator.py`](file:///Users/arvind/Documents/github/gemstack/src/gemstack/project/validator.py)
**Method:** `_extract_relevant_files()` (line 105–107)

1. Scope extraction to `## Relevant Files` section only (regex: `r"## Relevant Files\s*\n(.*?)(?=\n##|\Z)"`).
2. Filter out matches that:
   - Are single characters (`|`, `[`, `]`, `-`, `*`)
   - Start with markdown syntax (`#`, `**`)
   - Don't look like file paths (must contain `.` or `/`)
3. This eliminates the false-positive warnings that feed into the check/hook failure cascade.

### Task A4: Check `--strict` Flag

**File:** [`check_cmd.py`](file:///Users/arvind/Documents/github/gemstack/src/gemstack/cli/check_cmd.py)

1. Add `--strict` option: `strict: bool = typer.Option(False, "--strict", help="Treat warnings as errors")`.
2. When `--strict` is set AND there are warnings (but no errors), raise `ValidationError`.
3. Default behavior (no `--strict`) already exits 0 on warnings-only — this is correct.

### Task A5: Fix Hook Script Argument Syntax

**File:** [`hook_cmd.py`](file:///Users/arvind/Documents/github/gemstack/src/gemstack/cli/hook_cmd.py)

1. The pre-commit hook uses `gemstack check --project "$(git rev-parse --show-toplevel)"` but `check` takes a **positional** argument, not `--project`.
2. Fix to: `gemstack check "$(git rev-parse --show-toplevel)"`.
3. This is the root cause of hooks failing on every commit — `--project` is an unrecognized option.

## Phase B: Detection Improvements (separate PR)

### Task B1: Expand Python Dependency Sets

**File:** [`detector.py`](file:///Users/arvind/Documents/github/gemstack/src/gemstack/project/detector.py)

1. Add to `_BACKEND_PYTHON_DEPS`: `typer`, `click`, `fire` (CLI frameworks are backend topology).
2. Add to `_MLAI_PYTHON_DEPS`: `faster-whisper`, `ctranslate2`, `whisper`, `onnxruntime`.

### Task B2: Go `net/http` Backend Detection

**File:** [`detector.py`](file:///Users/arvind/Documents/github/gemstack/src/gemstack/project/detector.py)
**Method:** `_detect_go()` (lines 416–461)

1. After framework dep check, scan `.go` source files for `"net/http"` import.
2. If found, add `Topology.BACKEND` (unless already present).
3. Also detect HTML template files (`*.html` in `templates/` dir) as a frontend signal.

### Task B3: Go Library-SDK Heuristic Fix

**File:** [`detector.py`](file:///Users/arvind/Documents/github/gemstack/src/gemstack/project/detector.py)
**Method:** `_detect_go()` (lines 448–450)

1. Current logic: `pkg/ AND NOT cmd/` → library-sdk. This fails for SDK repos with example `cmd/` dirs.
2. New logic: if `pkg/` exists → add `Topology.LIBRARY_SDK` regardless of `cmd/`.
3. A project can have BOTH `library-sdk` and `backend` topologies (multi-topology is already supported).

### Task B4: Smart STATE Suggestion

**File:** [`validator.py`](file:///Users/arvind/Documents/github/gemstack/src/gemstack/project/validator.py)

1. When `[STATE: ...]` is missing, suggest a state in the warning message.
2. Heuristics (checked in order):
   - If STATUS.md contains "shipped" or "released" (case-insensitive) → suggest `SHIPPED`
   - If STATUS.md contains TODO/in-progress markers → suggest `IN_PROGRESS`
   - Default → suggest `INITIALIZED`
3. Warning message format: `"STATUS.md missing [STATE: ...] declaration (suggested: INITIALIZED)"`

## Phase C: Batch Operations (separate PR, large feature)

### Task C1: Project Registry

**New file:** `src/gemstack/cli/registry_cmd.py`

1. `gemstack registry add <path>` — register a project path in global config.
2. `gemstack registry list` — show all registered projects with health status.
3. `gemstack registry remove <path>` — unregister a project.
4. Registry stored in `~/.config/gemstack/registry.toml` via `platformdirs`.
5. Auto-discover: `gemstack registry scan <parent-dir>` — find all dirs with `.agent/`.

### Task C2: Batch Command Runner

**New file:** `src/gemstack/cli/batch_cmd.py`

1. `gemstack batch <command> [args...]` — run command across all registered projects.
2. Supported commands: `check`, `diff`, `migrate`, `hook install`.
3. Rich progress display with per-project status.
4. Summary table: project name, result (pass/fail), duration.
5. Exit code: 0 if all pass, 1 if any fail.

### Task C3: Register in CLI Main

**File:** `src/gemstack/cli/main.py`

1. Register `registry_app` and `batch_app` sub-commands.
2. Add help text linking to docs.
